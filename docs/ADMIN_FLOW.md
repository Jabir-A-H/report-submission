# Admin Flow

## Overview
This document outlines the workflows specific to Administrators. Admins have overarching control of the platform, user management, and city-wide data aggregation.

## 1. User Management & Approvals
Because the system relies on strict permissions, new registrations are gated.
- **Route**: `/admin/users`
- **Action / Approvals**: Admins view a list of users. If a user is pending approval (`active = false`), they display an **অপেক্ষমাণ** (Pending) badge. Admins can click the explicit green **অনুমোদন করুন** (Approve) action button to set `active = true` and grant access.
- **Suspensions**: Active users display a green **সক্রিয়** (Active) badge. Admins can revoke access at any time by clicking the explicit amber **নিষ্ক্রিয় করুন** (Deactivate) action button, which toggles `active = false` and prevents them from logging in.

## 2. Zone Management
- **Route**: `/admin/zones`
- **Action**: Admins create, edit, or delete geographical zones. Users are subsequently assigned to these zones.

## 3. Reviewing Zone Reports
- **Route**: `/admin/zone-reports`
- **Action**: Admins can impersonate or directly view any submitted report for any zone. Because Admins bypass RLS (Row Level Security), they have full read access to all entries.

## 4. City Report & Aggregation
The City Report is the core administrative feature, aggregating data from all zones.
- **Route**: `/admin/city-report`
- **Action**: Admins select a period. The system fetches the data dynamically generated by the 6 PostgreSQL Views.
- **Non-Monthly Aggregation**: For quarterly, half-yearly, nine-month, and yearly reports, the system automatically pulls the constituent monthly views and aggregates them using a client-side SUM logic to keep database views focused and reusable.
- **Exporting**: Admins can export the full City Report to customized PDF or Excel formats.

## 5. The Override System
Occasionally, mathematical aggregations must be manually adjusted for the official City Report.
- **Inline Editing (UX)**: Admins do not need to navigate to a separate, clunky management page. From the `/admin/city-report` view, they can click any aggregated number field to initiate an in-line correction.
- **Mechanism**: This saves a record to the `city_report_overrides` table (recording `year`, `month`, `section`, `field`, `value`).
- **Effect**: The PostgreSQL View or frontend aggregation instantly registers the override and serves the overridden value instead of the calculated sum. Overridden values are clearly highlighted in the UI so admins can see corrections at a glance.
